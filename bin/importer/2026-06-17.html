Auction Importer: Faster, Hands-Off Imports

A plain-language summary of an upgrade we're rolling out to the auction-import tool on caseantiques.com — so importing a large sale no longer ties up a staff member and an open browser tab.

Client: Case Antiques
Site: caseantiques.com
Prepared: June 17, 2026
Status: Building & deploying today

In one paragraph

Importing a large auction into the website — a sale can have 200+ lots, each with 15–30 photos — currently requires a staff member to open the Import screen and leave it open and untouched until the whole import finishes, which can take a long time. If that tab is closed, the computer goes to sleep, or the internet hiccups, the import stops partway. We're changing the importer so the work runs on the server, in the background: you pick the file, click Start, and can close the tab and walk away. The server finishes every lot and photo on its own, automatically retries anything that doesn't succeed the first time, and paces itself so it no longer strains the site. This work is being built and deployed today.

The problem

The current importer does all of its work inside your browser tab. As it runs, your open screen is literally what's driving the process — walking through each lot, then each of that lot's photos, one after another, loading every image into the website's Media Library.

For a big sale that means:

  • A staff member has to babysit the Import screen — leaving it open, awake, and connected — for the entire run.
  • If the tab is closed, the laptop sleeps, or the Wi-Fi drops, the import simply stops wherever it was.
  • There's no easy way to step away and let it finish, and no clean way to check on it from another computer.

It works for small batches, but it's slow and fragile for the large, photo-heavy sales that matter most.

Why it happens (the cause)

Two things are going on under the hood:

  • The browser is doing the heavy lifting. Because every step is driven from your open tab, the import can only go as fast as that one browser window — and it's only alive while the window is.
  • It asks the server to do too much at once. When it reaches a lot's photos, it fires off requests for many of them simultaneously. Processing a single photo is expensive — the site generates several resized copies of each one — so these pile up, some time out, and the tool retries them, adding even more load. (This is the same kind of strain involved in the recent morning slowdowns, so easing it helps the whole site.)

The upgrade

We're moving the import off the browser and onto the server's own background queue — the same kind of reliable, behind-the-scenes processing that already handles other automated tasks on the site. The change does three things:

ChangeWhat it does, in plain termsStatus
Runs in the backgroundYou pick the file and click Start. The website hands the whole job to the server and immediately tells you it's queued — you can close the tab and walk away. The server works through every lot and photo on its own time.Today
Retries automaticallyIf a photo or lot doesn't import on the first try, the server quietly tries it again later instead of giving up. Imports that used to stall now finish on their own.Today
Paces the workloadPhotos are processed steadily rather than in big bursts, so the import no longer strains the site for everyone else while it runs.Today

You'll still get a progress view — but now it's optional. Open it whenever you like (even from a different computer) to see how far along the import is; close it whenever you like without affecting the import.
