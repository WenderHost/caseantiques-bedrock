TL;DR. The DataTables thumbnail picks the "first" image with ORDER BY menu_order ASC LIMIT 1 and no unique tiebreaker. When the lowest menu_order is shared by 2+ images, MySQL returns an arbitrary one — and the LIMIT 1 plan can disagree with the full-list plan the admin media grid uses. 17 of 1,320 lots in 2026-summer have tied lowest menu_order; 4 are actively showing the wrong image right now.

1. The reported symptom

On the /auction/2026-summer/ DataTables view, Lot 274: Spanish Colonial Carved Santos Bust of Christ (post_id=765840) renders its thumbnail as: 

<img ... src="https://caseantiques.com/app/uploads/auctions/2026-summer/274_5-374x800.jpeg" ... alt="Lot 274: Spanish Colonial Carved Santos Bust of Christ">

Expectation was 274_2.jpeg — the image shown first in the admin Add Media → Uploaded to this post grid. The working assumption was that the API serves images by menu_order, set from the filename suffix at import (so 274_2 → menu_order = 2) or by manual reorder.

2. Root cause

2a. The query has no tiebreaker

The thumbnail comes from taxonomy.auction.php:271-278 (query_items_callback(), the wp_ajax_query_items handler that feeds DataTables): 

// lib/classes/taxonomy.auction.php — generated SQL:
SELECT wp_posts.ID FROM wp_posts
WHERE post_parent = 765840 AND post_type = 'attachment' ...
ORDER BY wp_posts.menu_order ASC
LIMIT 0, 1

There is no secondary sort key (no ID, no post_date). When the sort column ties, MySQL may return rows in any order, and that order can change with LIMIT because the query planner chooses a different access path.

2b. Lot 274's data was tied at zero (original state)

Note: this is the state at the start of the investigation. Lot 274 has since been manually reordered (§5), so its live data is now distinct 1–9 and the thumbnail is fixed. The snapshot below is preserved to document the original bug.

274_2, 274_7, 274_8, 274_4, 274_1, 274_6, 274_9, 274_50 (×8)20:38:13–16

Proven directly via wp eval — same ORDER BY, different first row:

  • posts_per_page => -1 (full scan) → returns 274_2 first  what the admin grid shows
  • posts_per_page => 1 (LIMIT 1) → returns 274_5 first  what the page shows
So the front-end thumbnail and the admin media grid disagree purely by tie-breaking luck. Nothing is cached or stale — the query is simply non-deterministic. Even the "expected" 274_2 is not backed by menu_order (it is 0, same as the rest); it only appears first in the modal by the same coincidence. If menu_order were repaired to follow the filename, the lead image would become 274_1 (→ 1), not 274_2.

2c. Why the menu_order is wrong (the importer)

auction-importer.php:909,921 assigns menu_order from the filename — but only when the attachment is newly created: 

// lib/classes/auction-importer.php (import_single_attachment)
preg_match( '/^([0-9]+[a-zA-Z]*)_([0-9]+)\.([a-zA-Z]+)/', $image, $matches );
if ( $attachment_id = $this->auction_object_exists( ... ) ) {
    $attachment['ID'] = $attachment_id;
    $update = true;                              // existing attachment -> menu_order NEVER set
} else {
    $attachment['menu_order'] = intval( $matches[2] );  // new only: 274_2 -> 2
}

If a lot is re-imported / partially replaced, existing attachments take the $update path and keep whatever menu_order they had (the default 0). Lot 274 fits this: 274_3 was inserted ~13s earlier and has the correct value (3), while the other 8 were left at 0.

5. Does the client's drag-and-drop reorder actually work? (workflow item #3)

The client's historical intent is: (1) name images {lot}_{seq}.(jpg|jpeg); (2) importer sets menu_order from {seq}; (3) client can drag-reorder images in the Add Media dialog. We tested whether #3 has ever taken effect.

Finding (corrected): drag-reorder in the Add Media dialog DOES work and persist. Confirmed on 16 lots / 74 reordered images in 2026-summer (filename-based scan), e.g. Lots 42, 44, 115, 126, 127, 133, 134, 162, 168, 173, 186, 192, 195, 274, 363 (plus Lot 15, which also has an odd-named file). Each has distinct menu_order values that do not match the filename {seq} — which can only come from a saved manual reorder.
How we initially got this wrong. Our first run of scan-reorder-evidence.php reported 0 hits and we briefly concluded reorder "never persisted." That was a bug in the script, not a fact about the data:
  • PHP casts a purely-numeric array key like "274" to the integer 274. So array_key_first($prefix_counts) returned int 274, while each image's parsed prefix was the string "274".
  • The dominant-prefix guard if ($p['prefix'] !== $dominant) continue; then compared "274" !== 274 with the strict operator → always true → every image was skipped, on every lot. Result: a false "0 hits."
  • Fix: compare as strings ((string)$p['prefix'] !== (string)$dominant). Re-run → 67 hits / 15 lots.
Lesson: a "0 results" diagnostic deserves the same scrutiny as a positive one. The empirical proof — the client reordering Lot 274 live and it sticking — is what exposed the bug.

Mechanism: the reorder fires WordPress core's save-attachment-order AJAX action, which renumbers all of the post's attachments to 1..N by drag position. That is why a reordered lot has clean, distinct, deterministic menu_order — and why reordering a previously-tied lot (like 274) instantly fixes its thumbnail.

Does query hardening override #3? No. The ID ASC tiebreaker is subordinate to menu_order — it only orders images that already tie. A real reorder sets distinct values, so the tiebreaker never fires and the client's order is honored exactly. Hardening and the reorder workflow are fully compatible.

8. Root cause of the zero menu_order (import)

Audit across the 8 most recent auctions (audit-auctions.php): zero-menu_order images appear in every auction back to 2023 at a steady low rate (5–175 images each, ~0.03–1.2%). Not a recent regression — if a script change caused it, older auctions would be clean. Two long-standing mechanisms, both confirmed:

#MechanismEvidence
1 Re-import zeroes it. The importer's update branch (auction-importer.php:916-918) calls wp_insert_attachment with no menu_order key. WP core wp_insert_post applies menu_order => 0 from defaults before the update check and never reloads the existing value — so reprocessing an existing image wipes its order to 0. The dev comment at :614-624 ("prevents menu_order from being overwritten") is backwards: omitting it causes the overwrite. post.php:4536 (defaults), 4561-4562 (update only reloads guid/status)
2 Regex-fail on first insert. menu_order = intval($matches[2]) from a regex requiring {digits}_{digits}.{ext}. LiveAuctioneers-id titles (9087223_10, no extension) don't match → intval('') = 0. 2026-summer: 44% of zero-mo images fail the regex vs 0.1% of non-zero

No post-import validation ensures every image has a sane, unique order. Implication: a future re-import can re-zero images — including lots staff hand-ordered — so the durable fix belongs in the importer (assign menu_order on the update path too / preserve it explicitly; add a validation pass), and an ongoing detect-and-fix report is worthwhile.
